# CIP-0004: VibeSafe Integration: User-Defined Checks and Tenet Layering

## Status

- [x] Proposed - Initial idea documented
- [ ] Accepted - Approved, ready to start work
- [ ] In Progress - Actively being implemented
- [ ] Implemented - Work complete, awaiting verification
- [ ] Closed - Verified and complete

## Summary

Define and implement the full integration between fynesse and VibeSafe, covering two mechanisms:

1. **User-defined checks**: A plugin mechanism in VibeSafe's `whats-next` script that discovers and runs domain-specific check scripts from `.vibesafe/checks/`. Fynesse provides its own `fynesse_checks.py` to report AAA-specific project status. This requires a change to VibeSafe (tracked separately below).

2. **Tenet layering**: Fynesse installs its tenets into `tenets/fynesse/` alongside VibeSafe's own `tenets/vibesafe/` and the user's `tenets/myproject/`, providing a clear three-layer tenet architecture.

## Motivation

When a user installs fynesse into a data science project, they should immediately have `./whats-next` telling them not just about CIPs and backlog (VibeSafe concerns) but also about their specific data science status: have the AAA stubs been implemented? Is there a legal/ethical basis documented in access.py? Is `machine.yml` present? Are there CIPs that describe the analysis design?

Without this integration, `whats-next` is silent on everything fynesse-specific. The user has to inspect files manually to know whether their access/assess/address implementation is complete.

The tenet layering question was: should fynesse tenets go into the target project? Yes — fynesse tenets explain WHY the AAA separation matters, and LLMs need these tenets visible in the same place as the project's own tenets so they inform decision-making.

## Detailed Description

### Part 1: User-Defined Checks in VibeSafe

**Discovery mechanism** (to be implemented in VibeSafe's `templates/scripts/whats_next.py`):

```python
def run_user_defined_checks(project_root):
    """Discover and run user-defined checks from .vibesafe/checks/."""
    checks_dir = project_root / ".vibesafe" / "checks"
    if not checks_dir.exists():
        return []
    
    results = []
    for check_file in sorted(checks_dir.glob("*.py")):
        try:
            spec = importlib.util.spec_from_file_location("check", check_file)
            module = importlib.util.module_from_spec(spec)
            spec.loader.exec_module(module)
            if hasattr(module, "run_checks"):
                check_results = module.run_checks(str(project_root))
                results.extend(check_results)
        except Exception as e:
            results.append({
                "level": "warning",
                "message": f"Check {check_file.name} failed: {e}",
                "suggestion": ""
            })
    return results
```

The function should be called from `whats_next.py`'s main output section, with results displayed after the core CIP/backlog/requirements output.

**Result format** (standardised dict):
```python
{
    "level": "ok" | "info" | "warning" | "error",
    "message": str,    # what was found
    "suggestion": str  # what to do (empty if level == "ok")
}
```

**Fynesse checks** (`templates/vibesafe-checks/fynesse_checks.py`, installed to `.vibesafe/checks/fynesse_checks.py`):

- `_check_aaa_modules`: are access.py, assess.py, address.py implemented or still stubs?
- `_check_machine_yml`: is machine.yml present for local configuration?
- `_check_legal_documentation`: does access.py have legal/ethical docs?
- `_check_assess_question_independence`: does assess.py illegally import address.py?
- `_check_cip_aaa_alignment`: are there CIPs that describe the analysis design?

### Part 2: Tenet Layering

Fynesse installs its tenets into `tenets/fynesse/` in the target project. The resulting structure is:

```
tenets/
├── vibesafe/          ← VibeSafe governance principles (installed by vibesafe)
│   └── *.md
├── fynesse/           ← Fynesse data science principles (installed by fynesse)
│   └── *.md
└── myproject/         ← Domain-specific tenets (written by the analyst)
    └── *.md           (optional, user-created)
```

The fynesse AI rules mention all three layers and encourage analysts to create their own domain tenets.

### How CIPs/Requirements/Backlog map to data science work

This mapping should be documented in the fynesse AI rules and in this project's documentation:

| VibeSafe artefact | Fynesse meaning |
|------------------|-----------------|
| Requirement (WHAT) | "Predict X from Y data with Z success criterion" |
| CIP (HOW) | "Access: Land Registry API; Assess: price distribution, postcode join; Address: linear model with these features" |
| Backlog task (DO) | "Implement access.data()" / "Write assess quality checks" / "Fit baseline model" |
| Tenet | Principles guiding all three phases (fynesse + vibesafe + project tenets) |

`whats-next` then shows: which analyses are designed (CIPs), which are in progress, what tasks remain, and (via fynesse checks) whether the code matches the design.

## Dependency on VibeSafe

The user-defined checks mechanism requires a change to VibeSafe's `whats-next` script. This should be raised as a CIP in the VibeSafe repository.

**VibeSafe changes needed:**
1. Add `run_user_defined_checks()` to `templates/scripts/whats_next.py`
2. Display check results in `whats-next` output with appropriate colouring
3. Document the check API in VibeSafe's README

Until VibeSafe supports this, the `fynesse_checks.py` file is installed but dormant. It can be run manually (`python .vibesafe/checks/fynesse_checks.py`) to see the check results.

## Backward Compatibility

The `.vibesafe/checks/` directory is new. VibeSafe ignores unknown directories, so installing `fynesse_checks.py` before VibeSafe supports the mechanism is harmless.

## Testing Strategy

- Implement access.py: verify stub warning disappears but legal-docs warning appears
- Add a license comment to access.py: verify all access checks pass
