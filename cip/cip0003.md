# CIP-0003: Canonical AI Rules for the Fynesse Framework

## Status

- [x] Proposed - Initial idea documented
- [ ] Accepted - Approved, ready to start work
- [ ] In Progress - Actively being implemented
- [ ] Implemented - Work complete, awaiting verification
- [ ] Closed - Verified and complete

## Summary

Design and create `rules/fynesse-framework.md` as the single canonical source of truth for AI assistant guidance on the Fynesse framework. This document is deployed by `install.sh` to Cursor, Claude Code, Codex, and GitHub Copilot. It covers the AAA module boundaries, common anti-patterns, the configuration pattern, and the workflow for working within a Fynesse project.

## Motivation

REQ-0006 requires that AI assistant rules guide LLMs to generate code that follows the AAA separation. Without these rules, LLMs will:
- Generate monolithic scripts that mix access, assessment, and addressing
- Add analysis-question-specific logic to the assess layer
- Bypass the config system by hardcoding paths and credentials
- Fail to document legal/ethical basis for data access

The rules must be derived from a single source to avoid drift between platforms. The explayner project demonstrates this pattern with `rules/explayner-architecture.md`.

## Detailed Description

### Document structure

`rules/fynesse-framework.md` covers:

**1. Framework overview** — what Fynesse is, what the three phases mean, why the separation matters

**2. Module boundaries** — for each of access, assess, address:
- What it must contain
- What it must NOT contain
- The key function signatures
- Brief correct/incorrect code examples

**3. Configuration pattern** — the `defaults.yml` → `machine.yml` → `_config.yml` layered config system; how to add new configuration keys; what must not be hardcoded

**4. Common anti-patterns** — with examples:
- Mixing access and addressing (SQL in address.py)
- Question-specific assessment (imputing for a model in assess.py)
- Skipping access (reading data directly in assess or address)
- Hardcoding credentials or paths
- Forgetting legal/ethical documentation in access

**5. Workflow** — how to add a new data source; how to add a new analysis; how to share the assess layer with collaborators

### Cursor rule deployment

The Cursor rule uses YAML frontmatter:
```yaml
---
description: Architecture rules for fynesse data science projects
globs:
  - "fynesse/**/*.py"
  - "fynesse/**/*.yml"
alwaysApply: false
---
```

Scoping to `fynesse/**` means the rule activates when editing module files but does not apply everywhere.

### CLAUDE.md / AGENTS.md deployment

Uses fenced markers so the section can be replaced on reinstall:
```
<!-- fynesse-framework -->
## Fynesse Framework

<content from rules/fynesse-framework.md>

<!-- fynesse-framework -->
```

### Design principles for the rules document

- **Concrete**: include short code snippets showing correct and incorrect patterns, not just abstract descriptions
- **Actionable**: tell the LLM what to do, not just what not to do
- **Concise**: the document should be readable in under 5 minutes; it must not become an encyclopedia
- **Self-contained**: the rules document should make sense to an LLM that has never seen the fynesse repo

## Implementation Plan

1. Write `rules/fynesse-framework.md` covering all five sections above

2. Test by running `install.sh` and verifying the rule appears correctly in Cursor, CLAUDE.md, and AGENTS.md

3. Review: ask an LLM to generate code for a simple data science task in a project with these rules installed, and verify the generated code respects the AAA boundaries

## Backward Compatibility

The fenced marker format (`<!-- fynesse-framework -->`) must be preserved across future versions of the rules so that reinstall can update the section.

## Testing Strategy

- Verify the generated `.mdc` file has correct YAML frontmatter and renders in Cursor
- Verify the CLAUDE.md section is inserted/replaced correctly
- Verify idempotency: reinstalling does not duplicate the section
- Manual review: read the rules and verify they are clear and actionable
