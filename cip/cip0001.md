# CIP-0001: Core Access-Assess-Address Module Architecture

## Status

- [x] Proposed - Initial idea documented
- [ ] Accepted - Approved, ready to start work
- [ ] In Progress - Actively being implemented
- [ ] Implemented - Work complete, awaiting verification
- [ ] Closed - Verified and complete

## Summary

Define the canonical module structure for a Fynesse project: three Python modules (`access.py`, `assess.py`, `address.py`) plus a configuration system (`config.py` / `defaults.yml`), with documented interfaces and clear responsibility boundaries. This structure becomes the template that is installed into target projects.

## Motivation

Without a canonical module structure, each analyst implements the AAA separation differently, making it impossible to share assess work across projects or for AI assistants to generate code that respects the framework. The structure must be simple enough to adopt immediately and rich enough to support real-world operational data science.

The `fynesse_template` repository already demonstrates this structure. This CIP canonises it within the fynesse framework project so it can be installed via `install.sh` and guided by AI assistant rules.

## Detailed Description

### Module structure

```
<project-root>/
└── fynesse/
    ├── access.py       ← data access (obtaining data, legal/ethical docs)
    ├── assess.py       ← data assessment (quality, structure, imports access)
    ├── address.py      ← question addressing (analysis, imports assess)
    ├── config.py       ← configuration loader
    └── defaults.yml    ← default configuration values
```

### `access.py` — the Access layer

Responsibility: obtain data from its source and return it in a structured format (typically a pandas DataFrame or equivalent). Must NOT contain quality checks, analysis logic, or anything that assumes a specific analysis question.

Mandatory: a comment or docstring documenting the legal/ethical basis for each data source accessed.

Key function signature: `access.data() -> DataFrame | None`

The access layer may also provide helper functions for specific data sources (e.g. `access.from_api()`, `access.from_database()`), but all of them are concerned with obtaining data, not understanding or analysing it.

### `assess.py` — the Assess layer

Responsibility: understand the properties of the data returned by access. This includes: encoding of missing values, data types, outliers, distributions, temporal coverage, and data quality flags. Must NOT contain analysis-question-specific logic.

Must import from `access` but must NOT make cleaning decisions that are specific to any downstream analysis question.

Key function signature: `assess.data() -> DataFrame | Any`

May also provide: `assess.query(data)` (human-in-the-loop quality check), `assess.view(data)` (visualisation of data properties), `assess.labelled(data)` (returns labelled subset for supervised learning tasks — labelling is a property of the data, not a question-specific operation).

### `address.py` — the Address layer

Responsibility: answer a specific analysis question using assessed data. This is where statistical modelling, machine learning, visualisation, and dashboarding live. May apply question-specific cleaning and feature engineering.

Key function signature: `address.analyze_data(data: DataFrame) -> dict | Any`

The address layer receives assessed data as input and does not call access functions directly (except in exceptional circumstances that must be documented).

### `config.py` — configuration loader

Loads configuration from a layered set of YAML files:
1. `fynesse/defaults.yml` — defaults checked into the repo (shared by all users)
2. `fynesse/machine.yml` — local overrides (gitignored, machine-specific)
3. `_config.yml` — project-level overrides (in repo root)

This pattern allows shared configuration for data URLs and project settings while keeping credentials and local paths out of the repository.

### Template stubs

The template stubs installed by `install.sh` (from `templates/fynesse/`) are minimal: they contain the correct imports, a module-level docstring explaining the layer's purpose, and stub implementations of the key functions with implementation guides. They are user content — the install script only copies them if they do not already exist.

## Implementation Plan

1. **Create template stubs** in `templates/fynesse/`:
   - `access.py` with module docstring, `data()` stub, error handling and logging pattern
   - `assess.py` with module docstring, `data()` stub that imports `access`, stub `query`/`view`/`labelled`
   - `address.py` with module docstring, `analyze_data()` stub
   - `config.py` with layered YAML loader
   - `defaults.yml` with minimal default configuration

2. **Document interfaces** in `rules/fynesse-framework.md` so AI assistants understand the boundaries

3. **Write tests** (optional for now) demonstrating the interfaces

## Backward Compatibility

This is an initial design; no backward compatibility concerns yet.

## Testing Strategy

- Verify template stubs import correctly with no data available (graceful failure)
- Verify assess.data() calls access.data() and handles None return
- Verify config.py loads from defaults.yml correctly
